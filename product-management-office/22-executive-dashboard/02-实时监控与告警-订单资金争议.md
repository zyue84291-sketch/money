# 02 — 实时监控（订单 / 资金 / 争议）

**用途**：**日更或实时** 经营与安全切片；**技术恢复步骤** 仍以 **`ops/RUNBOOK.md`**、**`spec`** 为准，本文件定义 **「看什么、谁看、什么算红」**。

---

## 1. 建议监控维度（占位）

| 域 | 典型指标 / 事件 | 红线索引 |
|----|-----------------|----------|
| **订单** | 创建率、支付成功率、完结率、取消原因分布 | 支付成功率 **连续 N 小时** 低于阈值 |
| **资金** | Escrow `deposit` / `release` 失败率、RPC 错误、对账差异 | 任何 **对账不平**、**大额异常挂单** |
| **争议** | 打开工单数、升级率、平均解决时长 | SLA 突破、**监管敏感词** 投诉激增 |

## 3. **城市异常指挥台（单屏字段 · 异常统一入口）**

**用途**：回答 **「现在这个城市什么状况？」** —— 风控 / Runbook / 升级 分散时，**指挥台** 强制 **同一行快照**（日更 **≥1 次**；**P0** 事件 **实时**）。数据源：**工单系统**（**`20/00` §2**）+ 经营看板（**`07`** / **`22/04`**）。

### 3.1 字段（**每城一行**）

| **字段** | **说明** | **示例（东京 JP-01 · 样板快照）** |
|-----------|----------|-------------------------------------|
| `city_id` | 与 **`12/00`** 一致 | `JP-01` |
| `as_of` | ISO 时间 | `2026-05-02T09:00+09:00` |
| `orders_open` | 进行中订单数 | **12** |
| `anomalies_open` | 内部标为 **异常** 的开放事件（含支付/索引/履约） | **3** |
| `payment_fail_open` | **`ticket_type=payment_failed`** 且未 `closed` | **2** |
| `dispute_open` | **`dispute_*`** 或 **`spec/03`** 流程中 | **1** |
| `on_call` | 当日 **Owner** | `CS-JP-Lead` |

### 3.2 纪律

- **不得** 用微信群计数代替工单；**§3.1** 与 **§3.3** 中 **工单类字段** 须与 **`20/00` §2.1** **`status` 枚举** 一致可对账。  
- **任一项 P0** → **15 min** 内更新 `as_of` 行 + 链 **`19/00`** 分工。

### 3.3 **运营指挥台（实时看板 · 今日）**

**用途**：值班长 **每班至少刷 1 次**（高峰 **30 min**）；数字 **必须** 能从 **工单系统 + 订单 DB** 自动或半自动导出（**禁止** 手填无依据）。

| **字段** | **定义（可 SQL）** | **东京样板** |
|-----------|-------------------|--------------|
| **`city_id`** | 与 **`12/00`** 一致 | `JP-01` |
| **`as_of`** | 快照时间 | `2026-05-02T18:00+09:00` |
| **`orders_today`** | **本自然日 0:00–as_of** 创建或达 **Escrowed** 的订单数（**城市维度**，与 BI 口径文档一致即可） | **12** |
| **`tickets_queue_open`** | `status` ∈ **`open` \| `triage` \| `assigned` \| `pending_user` \| `pending_internal`**（**`20/00` §2.1**）且 `city_id` 匹配 | **3** |
| **`payment_fail_open`** | `ticket_type = payment_failed` ∧ `status` ∉ `resolved,closed` | **2** |
| **`dispute_open`** | `ticket_type` ∈ `dispute_open`,`refund_dispute` ∧ `status` ∉ `resolved,closed`（或与 **`spec/03`** 状态机映射表） | **1** |
| **`orders_in_progress`** | 非终态订单数（与 **§3.1 `orders_open`** 可合并或并存，**二选一写死** 在 BI） | （与 §3.1 对齐） |
| **`on_call`** | 本班 Owner | `CS-JP-Lead` |

**与 §3.1**：若 **`orders_open`** 与 **`orders_today`** 并存，**周会** 固定解释 **「在途 vs 今日增量」**，避免 **重复叙事**。

### 3.4 **战场地图（v1.0 · 单一视图 · 每城一屏）**

**用途**：回答 **「现在是增长问题、风控问题，还是转化问题？」** —— **值班长 / 城市 Owner** **每班第一屏** 只看本节；**§3.1 / §3.3** 为 **字段字典**，**本节为「拼起来的一张图」**。

#### 3.4.1 今日城市状态（**一行读懂 · 东京样板**）

| **字段** | **含义** | **样板值** |
|-----------|----------|-------------|
| **`orders_today`** | 今日 **新建订单** 数（城市口径，与 BI 钉死） | **12** |
| **`escrowed_today`** | 今日 **进入 Escrowed**（或你们定义的托管态）笔数 | **5** |
| **`outreach_progress`** | **触达累计 / 周期目标**（默认 **W1 累计/周目标**，与 **`18/00` §6** 一致） | **28 / 40** |
| **`dispute_open`** | 开放争议工单数 | **2** |
| **`payment_fail_open`** | 开放支付失败工单数 | **1** |
| **`tickets_queue_open`** | 待处理工单队列（**`20/00` §2.1**） | （与 §3.3 对齐） |
| **`as_of` / `on_call`** | 快照时间 / 本班 Owner | 同 §3.3 |

#### 3.4.2 **战场主因（`battlefield_profile`）— 判定规则（防拍脑袋）**

| **优先级序** | **若成立（任一）** | **主因标签** | **今日默认叙事** |
|--------------|---------------------|--------------|-------------------|
| 1 | **`payment_fail_open` ≥1** 且 **Escrow 转化** 低于 **`15`** 黄线 **或** 大面积资金 incident | **`risk_funds`** | **先救火**：资金 / 支付 / 对账 → **暂停拉新加码**（见 **`19/00` §6**） |
| 2 | **`dispute_open` ≥2** **或** 监管 / 人身威胁工单 | **`risk_trust`** | **先风控与话术**：争议 / 欺诈 / 升级 → **KOL/投放让路** |
| 3 | **`outreach_progress` 落后 ≥30%** 且 **资金类工单 = 0** | **`growth_supply`** | **供给与触达**：BD / 社群 / 名单 → **客服不抢 BD 人力** |
| 4 | **`escrowed_today` / `orders_today` 显著低于** 近 7 日中位 | **`conversion`** | **转化与钱包教育**：`14` / 补贴合规 / 培训 → **增长动作降为 P2** |
| 5 | 以上皆弱 | **`balanced`** | 按 **`18/06`** 各角色 **默认任务** 执行 |

> **一线口诀**：**先看钱（支付/托管）→ 再看雷（争议/合规）→ 再看人（触达）→ 再看漏斗（转化）** —— 与 **`19/00` §6** 调度序一致。
