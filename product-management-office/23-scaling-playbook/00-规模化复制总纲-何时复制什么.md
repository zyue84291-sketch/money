# 00 — 规模化复制总纲（**何时复制 · v0.1 封包定义**）

**用途**：避免 **过早复制**（把失败放大）与 **过晚复制**（**单城英雄 ≠ 可融资 scale**）。

---

## 1. **三阶段门禁（数据化）**

| **阶段** | **进入条件** | **产出物** |
|----------|--------------|------------|
| **试点** | 单城、单供给池、明确 ICP；**`12/00` Gate** 书面 | **`18/00`** 执行日志 + **`07`** 基线埋点 |
| **验证** | 北极星与单位经济达 **`22`** / **`15`** **书面阈值** | 复盘纪要 + **失败模式清单** |
| **封包 v0.1** | **同一 Playbook** 由 **非原创始执行团队** 在 **第二城** 跑通 **最小成功标准**（见 **§3**） | **`23/01`～`03`** **定稿 v0.1** + **`99`** 证据索引 |

---

## 2. **「可复制」的定义（融资级）**

1. **SOP**：另一城市经理 **在不依赖创始人日常指挥** 下，能执行 **`18/00` 0→100 单** 子集并达标。  
2. **指标**：**`22/01`** 树上有 **叶子 + 数据源**；**`22/02` §3.4** 战场地图 **可日更**。  
3. **风险**：**`19`** 预案 + **`16`** 辖区清单 **已培训**。  
4. **Proof（本条为 VC 核心）**：**城市 A（创始密集）成功** + **城市 B（非原班）成功** —— **缺 B → 不可声称 scale**。

---

## 3. **封包 v0.1 —「第二城成功」最小标准（建议 · 可董事会改数字）**

> **默认第二城**：**SG-01**（与 **`12/00`**、**`18/00` §0** 一致）；若换城 **须书面替换**。

| **维度** | **城市 A（JP-01）** | **城市 B（SG-01）** | **验收** |
|----------|---------------------|---------------------|----------|
| **执行团队** | 可有创始成员 **≤30% FTE** | **非原班占比 ≥70%**（**`23/03`** 考勤定义） | HR 周报 + **`08/`** 登记 |
| **首 90 天** | 已达或路径清晰 | **Escrowed GMV ≥ Gate 下限 80%** 或 **书面豁免** | **`12/00` §3** |
| **工单与争议** | 已有基线 | **`dispute_rate` ≤ A 城同期 1.5×** | **`22/02`** |
| **Playbook 版本** | 母版 | **同一 `23/01` v0.1` git tag`** | 发布说明 |

**v0.1 封包动作**：`git tag playbook-v0.1` + Data Room **`99/evidence/`** 链接 **两城** 周报样本（脱敏）。

---

## 4. **反模式（路演禁止）**

- **「只有一城很火」** 且无 **Handover 证据** → **归为试点**，不得写 **「已验证复制」**。  
- **创始人亲自飞第二城救火** **≠** 失败，但须在 **`11/06` E1** 标注 **「过渡态」**。
