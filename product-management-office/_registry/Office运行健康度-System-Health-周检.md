# Office 运行健康度（System Health）— 周检

**用途**：**`22`** 监控 **业务指标**、**`15`** 监控 **实验**；本文监控 **PM Office 这套规则是否在跑** —— 否则会出现 **「指标好看、门禁全废」** 的 **纸上系统**。

**Owner**：**PM Owner**（无则产品负责人）；**频率**：**每周** 固定 **5～10 min**（建议挂在 **`22/03` 周会** 或 **`08/cadence`** 首栏）；**证据**：周纪要一行 + 下表数字。

**联动**：**[`执行检查清单-每日版.md`](执行检查清单-每日版.md)**（日）→ 本文（周汇总）；**[`违规记录.md`](违规记录.md)**；**[`../08-operations-pm/meetings/01-Meeting-OS-规则与会议绑定.md`](../08-operations-pm/meetings/01-Meeting-OS-规则与会议绑定.md)**（**§2** Meeting OS **执行率**）；**[`../08-operations-pm/audit/00-PM-System-Audit-Dashboard.md`](../08-operations-pm/audit/00-PM-System-Audit-Dashboard.md)** **M6**；超阈值 **系统修复** 走 **`21-product-operating-system`**（**`05`** 重排优先级、**`02`** 升级、**`06`** Incident）。

## 1. 周检指标（四类 + Meeting OS 可选栏）

| # | 指标 | 数据来源（示例） | 本周数 |
|---|------|-------------------|--------|
| A | **未通过 R1～R6 但已进入开发** 的需求条数 | Sprint / **`17`** 看板审计、PRD 链接 vs **`01` 附录 A** | |
| B | **未写 Impact Map（或三元组不全）却记为「方案评审通过」** 的次数 | 纪要 / **`违规记录.md`** / 工单 | |
| C | **本周新增 `违规记录.md` 行数**（含当场拒收、会后补记） | **`违规记录.md`** | |
| D | **`spec` 未更新却已对外宣称上线 / 扩大灰度** 的次数（含 **§2.4 满 48h 未补 spec**） | 发版记录、**`违规记录.md`**、**`spec/15` 附录〇** 对拍 | |
| **E** | **Meeting OS `compliance_rate`**（**`meetings/01` §2.0**） | 周会纪要链接 / Sheet；**阈值与动作** 以 **`meetings/01` §2.1** 为准 | |

**会议形式化信号（可选栏）**：若当周出现 **「应卡点未卡点」** 的口头举报，在纪要记 **1 行** 并计入 **C** 或触发 **§2**。**E** 与 A～D **并列上报**：**`audit/00` M6** 取 **E** 为 **Meeting OS** 真值；A～D 仍为 **文档/门禁绕过** 真值。

## 2. 阈值与分级（公司自填首版）

> 下列为 **启动用默认**；**首季** 由 **PM Owner + 工程负责人** 书面改成你们真实红线。

| 级别 | 条件（满足 **任一**） | 动作 |
|------|------------------------|------|
| **绿** | A=B=D=**0** 且 C ≤ **团队约定周上限**（默认 **2**，纯拒收类可不计入「系统坏」）；**若填 E** 则 E **不触发** **`meetings/01` §2.1** 黄/红阈 | 照常 |
| **黄** | A+B+D **合计 ≥ 1** 或 C **超过**周上限 **或**（已填 **E** 且触发 **`meetings/01` §2.1** 黄阈） | **当周**：主持人培训 **1 次**（**`meetings/01` §1.1**）；**PM Owner** 在 **`21`** 周栏 **点名整改 Owner + 日期** |
| **红** | A **或** D **≥ 1**（**绕门禁进开发 / 假上线**）**或**（已填 **E** 且触发 **`meetings/01` §2.1** 红阈） | **系统修复**：**48h 内** 开 **`21`** 专题（联席 **`01` §2.5** 执行人），输出 **补丁动作**（补流程 / 补工具 / 换人卡点）；必要时挂 **`21/06`** L1 |

## 3. 与「业务健康」的区别

| 维度 | **业务（`22`）** | **Office System Health（本文）** |
|------|------------------|-----------------------------------|
| 问句 | 收入 / 转化 / 留存好不好？ | **R1～R6、Impact、`spec`、会议门禁** 有没有被绕过？ |
| 若差 | 调增长、调产品 | **修操作系统**（规则、人、工具），否则业务数字 **不可信** |
