# 信任设计：评价、担保、UI

## 1. 三层信任（产品语言）

| 层级 | 手段 | `spec` 锚点 |
|------|------|-------------|
| **感知** | 向导 DID、历史单、媒体素材 | `spec/64`、`spec/30` |
| **交易** | Escrow、明确费用与终态、争议入口可见 | `spec/13` 资金区、`spec/01` |
| **事后** | 评价、申诉、客服 SLA | `spec/03` |

## 2. UI 纪律

- **资金区**：银行级克制；**`spec/13`**、**`spec/86`**。  
- **不得**用娱乐化动效掩盖 **金额/地址/网络**（`spec/13-1` 异常态）。

## 3. 首访信任路径（检查清单）

- [ ] 首屏 10 秒内说明 **「钱去哪、谁保管私钥」**  
- [ ] Escrow 页可解释 **「未完成 vs 已入金」**  
- [ ] 争议入口 **不隐藏**（但不误触）

## 4. 信任可量化（**Trust KPI · 与 `07/04` 同源**）

| 指标 | **定义** | **用于** |
|------|----------|----------|
| **`trust_score_proxy_a`** | `escrowed_sessions` / `payment_page_views` | 信任 + 支付摩擦合成 |
| **`trust_score_proxy_b`** | 点击「查看资金说明」后 **仍** `commit` 的比例 | 文案是否 **信息过载**（实验 **EXP-2026-0143**） |

**信任实验**：假设与 ICE 见 **`../07-metrics-growth/05-拟真实验全链路纸面推演×2.md`** 实验 2；**NSM** 见 **`../15-data-decision/00-北极星与分层指标.md` §1**。

## 5. **Escrow 心智验证（用户是否「理解托管」· 与录屏同窗）**

| **方法** | **通过标准（示例）** |
|----------|----------------------|
| **Moderated 任务**（`14/00` §4） | **≥70%** 受试者 **用自己的话** 说出「钱在合约/托管路径，**未完成前不归平台自由支配**」（措辞须 **`08-4`** 审过） |
| **错误信念捕获** | 若 **>30%** 认为「已打给向导 / 可随时退款无窗口」→ **Escrow 页** 必须改版（**`spec/13-1`** 资金区 + **`09/07`** 语义） |
| **与链上指标交叉** | **`trust_score_proxy`** 与 **Commit→Escrow** **同窗** 解释（**`07/04`**） |

**钱包 / approve 卡点**：定量口径放在 **`14/01` §3** 与 **`spec/06`**；本节只管 **「懂不懂」**。

## 与 spec/13-1 对齐检查（PR / 发版前 · **≥3 条**）

1. **评价、担保、资金说明** 模块是否对应 **`spec/13-1`** 已登记 **组件 / 区块**；**禁止**口头新增「银行托管」表述。  
2. **资金区** 版式、**异常态 Hint**、**争议入口** 是否与 **`13-1`** 及 **`spec/86`** 一致；**动效不得掩金额/地址/网络**（**`13-1` 异常态**）。  
3. 信任相关埋点（如 `trust_fund_flow_expand`）是否进入 **`spec/04` §3.4** 与 **`07/00`** 字典 **登记**。
