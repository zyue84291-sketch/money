# 01 — 决策优先级规则（增长 vs 风控 vs 收入 vs UX）

**用途**：把「谁说了算」变成 **可引用规则**；具体数字仍以 **`spec/`、`09-business-model`** 为准。

---

## 1. 决策类型四分法

| 类型 | 示例 | 默认裁决者 |
|------|------|------------|
| **A. 合规 / 资金 / 安全** | 提高 KYC、Escrow 异常处理 | **法务 + 工程 SSOT**；产品负责 **落地形态**，无权改链上真值 |
| **B. 商业 / 定价 / take rate** | 提高抽佣、打包增值 | **产品负责人 + 财务/商业**；须对齐 **`spec/08-4`**、**`09`** |
| **C. 增长 / 获客** | 补贴、渠道、降低注册门槛 | **增长负责人**；须在 **A** 允许范围内，并有 **`15`** 实验设计 |
| **D. UX / 转化** | 步骤合并、文案、动线 | **产品设计**；不得绕过 **A** 的必显信息与确认 |

---

## 2. 典型冲突矩阵（默认）

| 冲突 | 第一步 | 第二步（仍僵持） |
|------|--------|-------------------|
| 增长 ↓ 门槛 vs 风控 ↑ 审核 | 核对是否触碰 **Hard**（见 **`00` §1**） | **产品负责人** 召集 **法务 + 增长 + 数据**，48h 内出 **书面结论** → 记入 **`20/03`** |
| 商业 ↑ 抽佣 vs UX 顺滑 | **单位经济敏感性** 沙盘（**`09`**） | **产品负责人** 定 A/B 或分群定价；**`16`** 审对外口径 |
| 增长要新 API vs 工程要收口 | **R1～R6**（**`01-与工程-spec-SSOT-对齐契约`**） | 走 **`17/04`** 与 **`spec/07`** Wave 对齐 |

## 3. 「一票否决」

仅 **A 类** 可由 **法务或工程负责人（安全/链上）** 对发布行使 **veto**；须在 **`03`** 的 RACI 中登记。

## 4. 决策资源绑定（强制 · **落地能力**）

> **有决策 ≠ 能执行**：凡 **`21`** 联席 / 仲裁 / 周会 **`22/06`** 第四节产出的 **书面决议**，**必须** 同时满足下表；**缺一** → **不得进入执行**（**`17`** 不收录、**`15`** 不放大、**`12/05`** 不划拨人天），与 **`21/05` §0** 指标绑定 **合并检查**。

| 必填项 | 要求 |
|--------|------|
| **执行 Owner（唯一）** | **一个**姓名/账号；**禁止**「我们一起做」无主 Owner |
| **投入资源** | **人天** 和/或 **Squad 占用**（链 **`12-org-scaling/05`**、**`17-execution-roadmap`**）；若 **0 人天** 须写清 **「仅政策/流程、不改代码」** |
| **被暂停事项** | **必须列清单**（Epic / 实验 / 技术债 **标题或 ID**）；**若无暂停** 须显式写 **「无 —— 已对照 `12/05` 当季板无挤出」** |

**与 `21/05`**：量化排序与 **无效决策** 定义见 **`05`**；**本节** 为 **资源与 Owner** 的 **一票否决补全**。

## 5. 决策拆解（Execution Breakdown · **强制模板**）

> **有方向 ≠ 有动作**：会议结束若只有「共识」没有 **谁、在何时、交什么物**，**等于未执行**。凡 **`21`** 书面决议（含 **`22/06`** 第四节）**必须** 附下表；**缺表** → 与 **§4 缺一** 同等 —— **不得进入 `17` 排期**。

> **执行状态机（Execution State Machine）**：每条 Task **必须** 处于 **且仅处于** 下列 **四态之一**；**周会**（**`22/03` §0.2**）**强制更新** 本列，**禁止**「写了从未改状态」。

**复制到纪要 / `20/03` 同一条目下**：

| # | Task（可交付、可验收） | Owner（唯一） | 截止日期（YYYY-MM-DD） | **Status** |
|---|------------------------|----------------|------------------------|------------|
| 1 | | | | **Not Started** |
| 2 | | | | **Not Started** |
| 3 | | | | **Not Started** |

**Status 枚举（唯一合法值）**：**`Not Started`** · **`In Progress`** · **`Blocked`** · **`Done`**（大小写 **须与上表一致**，便于工单/API 同步）。

- **默认 ≥3 行**；若 **<3**，须 **书面豁免原因** + **产品负责人（或 `21/03` 仲裁）签字** 方可生效。  
- **Task** 禁止只写「推进 / 对齐 / 再看」；须 **动词 + 产出物**（例：「合并 PR #123」「更新 `spec/04` §3.4 一行」「发试点招募帖」）。  
- **Blocked** 须同时写 **阻塞物**（在 Task 文字后括号或同纪要 **一行**）；**Done** 须 **链证据**（PR 链接 / 发布记录 / 截图）。  
- **执行失败**（**Done** 但 **KPI 未改善**）→ **§5.1**（或 **`22/03` §0.5**）**Failure Loop**，**必须** 选 A/B/C 并写 **`20/03`**。

---

## 5.1 执行失败处理（Failure Loop · **战术层**）

> **与 `21/06` Incident** 区分：**`21/06`** 管 **事故 / 暴跌战情**；本节管 **「活干完了，数没动」** 的 **常规纠偏**。

**若** 某条 Task 已为 **`Done`**，但其绑定的 **`22/04` 叶子** 在 **`21/07` §4** 验证窗内 **未达预期 / 无改善** —— **48h 内** 须 **书面三选一**（记入 **`20/03`** + 周会 **§0.5** 复读）：

| 选项 | 含义 |
|------|------|
| **A · Continue** | **继续**：**加资源** 或 **延长验证窗**（须写 **新人天 / 新截止日期**） |
| **B · Adjust** | **调整**：**改方案** 并 **新开 ≥2 条** 替换 Task（**Status** 重置为 **Not Started**） |
| **C · Stop（Kill）** | **停止**：该路径 **Kill**；释放 **`21/01` §4** 占用；必要时触 **`21/00` §5** / **`01-strategy/02`** |

**升级**：若 **A/B 两轮** 后仍无改善 → **升级 `21/02` L3** 或按 **`21/06`** 阈值进 **L2/L1**。
