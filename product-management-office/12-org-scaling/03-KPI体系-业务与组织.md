# KPI 体系：业务与组织（非仅技术指标）

## 1. 分层 KPI（示例结构）

| 层级 | 示例 KPI | 数据源 |
|------|----------|--------|
| **公司** | Runway、月度净收入、现金消耗 | 财务 |
| **增长** | 新客 CAC、邀请 K-factor、渠道 ROI | **`07-metrics-growth`** + 渠道表 |
| **交易** | GMV、入金率、完成率、争议率 | 链上事件 + DB 投影 `spec/01` |
| **供给** | 活跃向导、审核通过率、接单时延 | 运营 + API |
| **技术** | 可用性、P0 事故、索引延迟 | SRE、`spec/120` |

## 2. 与技术指标的边界

- **「索引器落后 X 分钟」** 是 **SRE KPI**，**不能**单独证明商业成功。  
- 董事会级应至少看到 **GMV / 收入 / 供给密度** 之一。

## 5. **客服与升级 SLA（数字 · v0.9）**

**定义**：**L1** = 一线客服 **首次触达用户**；**L2** = 值班组长 / 城市运营 **给出方案或技术确认**；**L3** = 管理层或 **on-call** 书面裁决（可链 **`21/02`**）。**开单时** 必须写 **`sla_tier`**（与 **`20/00` §2** 一致）。

| **层级** | **时钟** | **适用** |
|----------|----------|----------|
| **L1** | **10 分钟内** 首次回复（工作时段按城市日历；**P0** 7×24） | 所有 **`priority=P0`**；**支付 / 资金 / 安全** |
| **L2** | **2 小时内** 更新（含内部排查结论或「仍需延长」说明） | **P1** 默认；**P0** 若未结案则 **每 2h** 同步直至降级 |
| **L3** | **24 小时内** 书面结论（邮件或工单系统 @用户） | **P2/P3** 首响；**跨部门/法务/退款模板外** |

**判责**：**突破 SLA** → 记入 **`22/05`** 问责表 + **`08/`** 周会 **红项**；**连续 3 次** L1 违约 → **CS Lead** 须输出根因（人力 / 工具 / 口径）。

## 6. **压力测试下的 KPI（洪峰模式）**

| **模式** | **临时顶线（示例）** | **解除条件** |
|----------|----------------------|----------------|
| **dispute 队列深度** | **>20** 并行 → **全员客服支援** 触发（**`12/README`**） | **<10** 持续 **24h** |
| **P0 支付** | **任意 1 起** → **L3 每 2h** 同步至 **`22/02`** | **退出 Runbook** |

**与 UX/Growth**：洪峰后 **7 日** 必须复盘 **`14/01` §3`**（转化是否被客服拖垮）。
