# 争议处理 SOP（运营视角）

## 1. 与协议的关系

- **规则真值**：`spec/03`、`spec/01` 状态机、仲裁与执行器 **`spec/01` §7**。  
- **本 SOP**：**用户如何提交证据、时效、客服话术、何时升级仲裁、如何同步用户「未放款」**。

## 2. 工单字段（**争议类 = 基线 + 扩展**）

**基线（必填）**：**`../20-product-iteration/00-用户反馈收集与闭环.md` §2**（`ticket_id`、`ticket_type`、`priority`、`sla_tier`、`owner`、`status`、`city_id`、`order_id` 等）。**`ticket_type`** 建议使用 **`dispute_open`** / **`refund_dispute`** 等于 **`19/05`** 枚举对齐。

**争议扩展（在基线之上追加）**：

| 字段 | 说明 |
|------|------|
| `on_chain_status` | 只读展示 |
| `dispute_round` | 与 dispute 多次规则 `spec/01` §10 对齐 |
| `evidence_pack_url` | 大小与类型白名单 `spec/01` |
| `escalation_reason` | 升级原因 |

## 3. 与 **16** 边界

- 涉 **跨境证据合法性** → **法务**（`16`）。

---

## 4. **每类用户话术（争议场景）**

| **用户类型** | **用户常见说法** | **L1 标准口径（摘要）** | **详稿** |
|--------------|------------------|-------------------------|----------|
| **Traveler** | 「我要退全款 / 对方违约」 | 「退款与释放以 **`spec/03`** 与链上状态为准；我为您开立争议单并说明 **证据截止日**。」 | **`05` 话术大全 §1/§3** `refund_dispute` |
| **Guide** | 「对方恶意差评 / 我已完成服务」 | 「我们会 **双边取证**；请上传 **可验证时间线**。裁决前 **单方说法** 不对外承诺。」 | **`05` §3** + 本 §2 `evidence_pack_url` |
| **双方** | 「为什么还没打款」 | 「放款由 **执行器 / 仲裁结果** 驱动；客服 **无权** 按键放款。」 | **`19/00` §2** 分工 |

---

## 5. **与 Runbook 分工（争议类）**

| **Runbook / 工程** | **`19` 运营** |
|--------------------|---------------|
| **合约状态查询、执行器 retry、索引修复** 导致「应释放未释放」 | **告知用户「在排查」** + 工单挂 **`engineering_ticket`** |
| **仲裁合约入口、证据哈希上链**（若产品化） | **指导用户提交材料**、**不得**代填虚假证据 |

---

## 6. **SLA（争议工单）**

| **节点** | **时钟** | **备注** |
|----------|----------|----------|
| **受理开单** | **L1 ≤10 min**（P0）或 **≤24h**（P2，**`12/03` §5**） | `ticket_type` 必选 **`dispute_open`** 或 **`refund_dispute`** |
| **证据清单送达用户** | **≤24h** 内书面 | 与 **`spec/01`** 窗口一致时以 **协议** 为准 |
| **升级仲裁 / 工程** | **≤2h** 内完成 **内部分派**（Owner 写入工单） | 同步用户在 **L2** 窗口 |
