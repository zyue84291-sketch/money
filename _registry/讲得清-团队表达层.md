# 讲得清 — PM Office 给**全员**的表达层

**用途**：Office **逻辑完整**，但 **名词多、目录深** —— 本页把 **同一套规则** 翻成 **团队能复述的话**；**不替代** **`01`/`02`/`21`** 正文，用于 **周会开场、新人宣贯、跨部门对齐**。  
**谁维护**：**PM Owner**；**更新**：规则改名或增删门禁时 **同步改本页**（控制在 **1 屏内**）。

---

## 0. 我们只看三件事（**全员背这三句**）

1. **写需求** → 先过 **自检（R1～R6）** + 贴 **`新需求-最小入口-模板`**（**禁止**「口头说、会上再填」进需求评审）。  
2. **改东西** → 写 **Impact**（**我会影响谁**；跨模块写满 **`02`** 三元组再进方案评审）。  
3. **有冲突** → 上 **`21`**（谁拍板、占多少资源）。

> **讲不清这三句 = 团队一定不会用** —— 周会开场 **先复述 30 秒**，再谈具体需求。

---

## 1. 三句话：我们到底在干什么？

1. **产品写清「要什么」**，工程在 **`spec/`** 写清 **「真值是什么」** —— **两套纸**，**不能打架**；打架时 **以 `spec/` + 财务可审计为准**。  
2. **改 anything 会连累别人**（支付、运营话术、增长实验）→ 必须先写 **「我从哪来数据 / 我会影响谁」**（**Impact**），**再**进方案评审。  
3. **会上说的要落地**：该拒收就拒收、该记就记 **`违规记录`**；**文档、会议、报表** 都不能代替 **上线证据和钱是否真进来**。

---

## 2. 名词 → 人话（开会只许用右列）

| 文档里写的 | 团队可以这样说 |
|------------|----------------|
| **R1～R6** | **PRD 自检条**：每条需求有没有 **对上 spec 章节**、有没有 **踩资金雷**、有没有写清 **上线怎么验** |
| **Impact Map / 三元组** | **「我一动，谁会疼」**：数据从哪来、会打到哪些模块、最坏会怎样 |
| **`21`** | **拍板顺序**：增长 / 风控 / 钱 / 体验 **打架时谁先说、谁有一票否** |
| **`22`** | **经营看板**：老板周会看的 **北极星 + 红绿线** |
| **Gray / §2.4** | **spec 还没写但必须先试**：PRD 里写清 **临时假设**，登记，**48 小时内** 补 spec |
| **System Health 周检** | **门禁有没有形同虚设**：有没有人 **没过自检就开发**、有没有 **假上线** |

---

## 3. 对工程 / 设计 / 运营各一句

- **工程**：你们只认 **`spec/`** 和门禁；**产品不得**在 PRD 里 **发明状态机 / API / 资金路径** —— 要改走 **评审 + 改 spec**。  
- **设计 / 增长**：可以大胆试，但 **动漏斗、动钱、动链上相关文案** → **必须** 过 **Impact + 对应会议**（见 **`../08-operations-pm/meetings/01-Meeting-OS-规则与会议绑定.md`**）。  
- **运营 / 客服**：对外口径 **只跟 `spec` + Runbook**；**Office** 帮你们对齐 **「不能乱承诺什么」**，见 **`权限边界-强执行.md`**。
