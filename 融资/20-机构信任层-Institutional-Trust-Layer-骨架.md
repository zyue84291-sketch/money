## 1. Security Audit（安全审计）

**要写清**：**审计范围**（合约 / 链下 / 密钥流程）、**厂商**、**报告日期**、**严重项闭环状态**、**复审计节奏**。

- [ ] 报告与 **`07`** 中版本/Commit **可对应**  
- [ ] **已知未修复项** 有 **风险接受记录**（董事会或技术负责人）  
- [ ] **赏金计划 / 渗透**（若有）索引在 `14` Security；**流程与 runbook** 见 **`38`**  

---

## 2. Legal Opinion（法律意见书）

**要写清**：**司法辖区**、**问题清单**（Token 属性、业务模式、托管）、**结论摘要** vs **全文 Off-repo**。

- [ ] 对外 **只引用结论边界**，不 **过度承诺**  
- [ ] 与 **`06` 监管矩阵** **同一法律顾问口径**  

---

## 3. Financial Controls（财务内控）

**要写清**：**认收入政策**、**审批流**、**银行签字权**、**月结清单**、**是否与链上数据对账**。

- [ ] **管理报表 → 投资人月报** 路径单一（与 `17`、`02` 一致）  
- [ ] **审计师**（若有）与 **未审计区间** 对外表述 **分离**  

---

## 4. 第三方合作背书（Credibility markers）

**要写清**：**知名客户 POC**、**审计/律所/托管品牌**、**大学或开源社区**、**行业协会**——每项：**合同状态**、**可披露级别**、**联系人是否可 DD**。

- [ ] **Logo wall** 与 **真实授权** 一致（避免 Deck 侵权）  
- [ ] **`25` 生态伙伴** 与本节 **去重或交叉引用**  

---

## 5. 信任层「发布门禁」

- [ ] 新增任一背书前，过 **`09`** + 法务 **一句话可用口径**  
- [ ] **过期材料**（旧审计、旧 opinion）从 Data Room **撤下或标注 superseded**
