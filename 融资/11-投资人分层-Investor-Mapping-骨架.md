## 1. 投资人画像（Personas）

为每一类账户写 **「他们用什么语言评价你」**（指标、合规、协议、GMV、品牌）。

| 类型 | 典型关注点 | 你方对应材料入口 |
|------|------------|------------------|
| Web3 / Crypto 原生基金 | 协议设计、代币经济学、链上指标、安全、监管口径 | `07`、`13`、`15` |
| Travel Tech / OTA 产业 | 供给、转化、复购、线下运营、单位经济 | `02`、`08` |
| AI 基金 | 模型/数据飞轮、护城河、成本曲线 | `03`、`07` |
| 战略投资方（Corp Dev） | 协同、排他、数据、市场进入 | `05`、`25` |
| 家办（Family Office） | 下行保护、条款简洁、叙事信任、节奏 | `10`、`16` |

- [ ] **每类至少 3 家** 已建档（见 §5），非泛泛「Web3 都见」  

---

## 2. 基金偏好（Mandate Fit）

**每张卡片**：基金名、**阶段**、**单笔 check**、**地理**、**已投竞品/邻域**、**禁忌**（某些机构不碰 Token）。

- [ ] **竞品投资组合** 已标红（避免利益冲突未披露）  
- [ ] **Lead 意愿**（能写 term / 能带 syndicate）与 **跟投** 分开维护  

---

## 3. 投资阶段（Stage）

**要写清**：对方基金当前 vintage、本赛道 **最近一笔** 公开轮次（辅助判断 timing）。

- [ ] 本项目的 **真实阶段**（产品/收入/链上）与对外 **轮次命名** 一致  

---

## 4. 投资逻辑匹配（Message-Mandate Matrix）

**矩阵**：行 = 我方支柱（例如：旅行 RWA、协议、AI）；列 = 基金类型；格内 = **一句话 hook + 要推的数据**。

- [ ] **同一 Deck** 是否需 **2～3 个版本**（Travel-heavy / Protocol-heavy）已决策，避免现场改故事  

---

## 5. 优先级名单（Target List）

**分层**：A（必攻，强匹配）、B（补充 syndicate）、C（长周期 nurture）、**暂不接触**（含原因）。

| 字段 | 说明 |
|------|------|
| 基金 / Partner | |
| 引入人（warm intro） | |
| 上次动作 / 下次动作 | 与 `12` CRM 同步 |
| 状态 | 未触达 / 材料已发 / IC 前 / DD / Pass 原因 |

- [ ] **每周 Review** 负责人已定  
- [ ] Pass 原因 **结构化记录**（阶段 / 赛道 / 条款 / 其他），用于迭代叙事
